A Re-occurring Situation

  • An object wants to perform a request: Sender
  • The request can have more than one potential handler: Receiver(s)

Constraints:

  • The request will eventually be handled by one receiver.
  • Each handler can decide to either:
    • Perform the query or command and end further processing of the request.
    • Reject the request and let other handlers process it.
  • Handlers process the request sequentially.

Naïve Approach

  • A series of function calls.
  • Cons:
    • Hard coupling between the sender and the receivers.
    • Can't change handlers or their order at runtime.

Attempts at a Solution

                                        
public interface Handler
{
    void handle(Context context);
}

public class Client
{
    public Client(private List<Handler> handlers) {}

    public void sendRequest(String uri)
    {
        HttpContext context = createHttpContext(uri);

        foreach(var handler in handlers)
        {
            if(handler.handle(context) == BREAK)
            {
                break;
            }
        }
    }
}

A Dynamic Array

  • We store handlers in a dynamic array.
  • Pros:
    • Handler interface: Decouples the sender from the receivers.
    • Add and remove handlers at runtime.
  • Cons:
    • Mediocre to low Performance.
    • Insertion and deletion are expensive
      operations: Memory reallocations.

Solution

                                        
public interface Handler
{
    void handle(Context context);
}

public abstract class BaseHandler implements Handler
{
    private next: Handler;

    void handle(Context context);
    Handler setNext(Handler handler);
}

public class ConcreteHandler : BaseHandler
{
    void handle(Context context)
    {
        if(canHandler(context))
        {
            //Process the request and end further processing...
        }
        else
        {
            //Pass the request down the chain...
            next.handle(context);
        }
    }

    Handler setNext(Handler handler)
    {
        next = handler;
    }
}

public class Client
{
    public Client(private Handler firstHandler) {}

    public void sendRequest(string uri)
    {
        HttpContext context = createHttpContext(uri);
        firstHandler.handle(context);
    }
}

Benefits

  • Single Responsibility Principle
    • Decouple the sender of a request from its receiver(s).
  • Open/Closed Principle
    • Introduce new handlers without breaking existing code.
    • Add or change responsibilities by adding to or changing the chain at runtime.

Limitations

  • A request has no explicit receiver.
  • There’s no guarantee it’ll be handled.
  • The request can fall off the end of the chain.

Observer

Intent

Observer is a behavioral design pattern that lets you define a subscription mechanism to notify multiple objects about any events that happen to the object they're observing.

Intent

It defines a one-to-many dependency between objects so that when the one object changes state, all its dependents are notified and updated automatically.

Structure

  • The oberserved object notifies the subscribers when an event happens (its state changes).
  • The observed object knows all of the observers.
We'll call it the "Observable".

Structure

Structure of Observable

Structure

  • The observing objects are completely independant.
  • However, they share the same functionality : react when notified by the object they're observing.
  • They must all implement a shared interface.
We'll call them the "Observers".

Real World Applications

GUI Event Listeners

Buttons, sliders, or text fields notify multiple listeners when their state changes (a button click triggers multiple UI updates, for example).

Game Development

Entities such as enemies observe player actions; when the player moves or attacks, observers react accordingly.

Real-Time Logging Systems

Various logging outputs (such as the console or files) subscribe to application events to record them in real-time.

import java.util.ArrayList;
import java.util.List;

public class Stock {
    private List<Observer> subscribers = new ArrayList<>();
    private String name;
    private float price;

    public Stock(String name, float price) {
        this.name = name;
        this.price = price;
    }

    public void subscribe(Observer obs) {
        subscribers.add(obs);
    }

    public void unsubscribe(Observer obs) {
        subscribers.remove(obs);
    }

    public void notifySubscribers() {
        for (Observer subscriber : subscribers) {
            subscriber.update(this);
        }
    }

    public void setPrice(float newPrice) {
        // Notify the observers only if the price changes
        if (this.price != newPrice) {
            this.price = newPrice;
            notifySubscribers();
        }
    }

    public float getPrice() {
        return this.price;
    }
}

public interface Observer {
    public void update(Stock context);
}

public class TableFacade implements Observer {
    private Table table;

    @Override
    public void update(Stock context) {
        // Update the Table
        table.updateTable(context);
    }
}

public class NotificationFacade implements Observer {
    private Notifier notifier;

    @Override
    public void update(Stock context) {
        // Send a notification if a certain limit is exceeded
        if (context.getPrice() >= PRICE_CEILING) {
            notifier.sendNotification();
        }
    }
}

public class GraphingFacade implements Observer {
    private Graph graph;

    @Override
    public void update(Stock context) {
        // Update the graph
        graph.updateGraph(context);
    }
}

import okhttp3.*;

public class Main {
    static Stock stock = new Stock("StockA", 50.4f);

    public static void setupAPI() {
        OkHttpClient client = new OkHttpClient();

        // Building the request
        Request request = new Request.Builder()
                .url("https://realstock.com/api")
                .build();

        // Defining the listener's behavior
        EventSourceListener listener = new EventSourceListener() {
            @Override
            public void onEvent(EventSource source, String id, String type, String data) {
                float newPrice = Float.parseFloat(data);
                stock.setPrice(newPrice);
            }
        };

        // Build the source, send the request and link it to the listener
        EventSources.createFactory(client).newEventSource(request, listener);
    }

    public static void main(String[] args) {
        // Setup observers
        TableFacade tableFacade = new TableFacade(new Table());
        NotificationFacade notificationFacade = new NotificationFacade(new Notifier());
        GraphFacade graphFacade = new GraphFacade(new Graph());

        stock.subscribe(tableFacade);
        stock.subscribe(notificationFacade);
        stock.subscribe(graphFacade);

        setupAPI();

        Application app = new Application();
        app.run();
    }
}

Implementation of the Observers

Benefits

  • Open/Closed Principle
    • You can introduce new observer classes without having to change the observable's code.
  • Single Responsibility Principle
    • Each observer handles its own reaction to the update.
    • The observable only manages state and notifications.

Limitations

  • Observers are notified in random order.
  • The observable doesn’t know its observers, but their reactions might indirectly depend on it.
