Another situation...

Requirements

  • Multi-process architecture.
  • A bunch of handlers linked up by queues.
  • Each handler must receive the request.

Proposed Solution

Using Chain of Responsibility

Proposed Solution

  • The order in which the handlers are placed doesn't matter.
  • The handlers are tasked with forwarding the request and handling it, breaking the single responsibility principle.
  • We can decouple these two responsibilities.

Proposed Solution

The intent is completely different from Chain of Responsibility!

Observer

Intent

Observer is a behavioral design pattern that lets you define a subscription mechanism to notify multiple objects about any events that happen to the object they're observing.

Intent

It defines a one-to-many dependency between objects so that when one object changes state, all its dependents are notified and updated automatically.

Structure

  • The oberserved object notifies the subscribers when an event happens (its state changes).
  • The obversed object must contain a reference to all of them.
We'll call it the "Observable".

Structure

Structure of Observable

Structure

  • The observing objects are completely independant.
  • However, they share the same functionality : react when notified by the object they're observing.
  • They must all implement a shared interface.
We'll call them the "Observers".
