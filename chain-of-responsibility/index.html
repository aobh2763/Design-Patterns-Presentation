A Re-occurring Situation

  • An object wants to perform a request: Sender
  • The request can have more than one handler: Receiver(s)
  • Ideally, the sender should not be coupled with all the potential receivers.

Constraints:

  • Handlers process the request sequentially.
  • Each handler can decide to either:
    • Perform the query or command and end further processing of the request.
    • Pass the request to the next handler.

Naïve Approach

  • A series of function calls.
  • Cons:
    • Hard coupling between the sender and the receivers.
    • Can't change handlers or their order at runtime.

Attempts at a Solution

                                        
public interface Handler
{
    void handle(Context context);
}

public class Client
{
    public Client(private List<Handler> handlers) {}

    public void sendRequest(String uri)
    {
        HttpContext context = createHttpContext(uri);

        foreach(var handler in handlers)
        {
            if(handler.handle(context) == BREAK)
            {
                break;
            }
        }
    }
}

A Dynamic Array

  • We store handlers in a dynamic array.
  • Pros:
    • Handler interface: Decouples the sender from the receivers.
    • Add and remove handlers at runtime.
  • Cons:
    • Mediocre to low Performance.
    • Insertion and deletion are expensive operations: Memory reallocations.

Solution

                                        
public interface Handler
{
    void handle(Context context);
}

public abstract class BaseHandler implements Handler
{
    private next: Handler;

    void handle(Context context);
    Handler setNext(Handler handler);
}

public class ConcreteHandler : BaseHandler
{
    void handle(Context context)
    {
        if(canHandler(context))
        {
            //Process the request and end further processing...
        }
        else
        {
            //Pass the request down the chain...
            next.handle(context);
        }
    }

    Handler setNext(Handler handler)
    {
        next = handler;
    }
}

public class Client
{
    public Client(private Handler firstHandler) {}

    public void sendRequest(string uri)
    {
        HttpContext context = createHttpContext(uri);
        firstHandler.handle(context);
    }
}

Chain Of Responsibility

  • We use a Linked List.
  • Pros:
    • Handler interface: Decouples the sender from the receivers.
    • Add and remove handlers at runtime.
    • Optimal performance.

Decorator vs CoR

  • The same class diagram.
  • But they differ in intent:
    • A Decorator adds functionality to an existing object.
    • In a CoR, a Handler has two roles:
      • Process the request.
      • Determine if it needs further processing.

{
    app.UseExceptionHandler("/Home/Error");

Benefits

  • Single Responsibility Principle
    • Decouple the sender of a request from its receiver(s).
  • Open/Closed Principle
    • Introduce new handlers without breaking existing code.
    • Add or change responsibilities by adding to or changing the chain at runtime.

Limitations

  • A request has no explicit receiver.
  • There’s no guarantee it’ll be handled.
  • The request can fall off the end of the chain.
